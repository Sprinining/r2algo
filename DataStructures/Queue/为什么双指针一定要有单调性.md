## 为什么双指针一定要有单调性

### 1. 原始代码与翻车现场

以下是基于滑动窗口编写的“寻找和至少为 K 的最短子数组”的代码。这段代码在**全为正数**的数组中运行完美，但面对**包含负数**的数组时会逻辑失效。

```c
#define MMIN(a, b) ((a) > (b) ? (b) : (a))
#define INF_MAX 0x3f3f3f3f

int shortestSubarray(int* nums, int numsSize, int k) {
    int res = INF_MAX;
    int sum = 0;

    for (int l = 0, r = 0; r < numsSize; ++r) {
        sum += nums[r];
        
        // 【为什么全为正数时代码正确？】：
        // 1. 单调递增：r 向右移动时，sum 一定会变大（或不变）。
        // 2. 单调递减：l 向右移动时，sum 一定会变小。
        // 
        // 当 sum >= k 成立时，因为全是正数，当前窗口 [l, r] 就是以 r 结尾的
        // 候选解。此时为了找“最短”子数组，我们必须尝试把 l 向右移（缩小窗口）。
        // 只要 sum 依然 >= k，说明缩小的窗口仍然合法，继续缩，直到不满足为止。
        // 这种单调性保证了 while 循环能精确榨干当前 r 下的所有更短可能性。
        //
        // -----------------------------------------------------------------
        // 【为什么有负数时会错？反例模拟】：
        // 设 nums = [84, -37, 32, 40, 95], k = 167 (正确答案是后3项, 长度为3)
        // 
        // 1. 右指针 r 走到末尾(r=4)时，把所有数都加了进来，此时 sum = 214。
        // 2. 满足 sum >= k (214 >= 167)，代码第一次进入下面的 while 循环：
        //    - 更新 res = MMIN(INF_MAX, 4 - 0 + 1) = 5
        //    - 减去左边界 nums[0](84)，sum 变成 214 - 84 = 130
        //    - 执行 ++l，此时 l 变成 1 (指向 -37)
        // 3. 准备进行下一次 while 循环条件检查：
        //    - 检查 sum >= k (130 >= 167)，发现不满足！
        //    - while 循环直接退出，外层 for 循环也结束了。
        //
        // 【致命问题】：
        // 此时 l 停在 1。如果 l 能继续往右移动到 2 (移出负数 -37)，
        // 那么 sum 会变成 130 - (-37) = 167，再次满足 >= k 的条件！
        // 这样就能把 res 更新为 4 - 2 + 1 = 3 (找到更短的正确答案)。
        // 
        // 也就是说，由于负数 -37 的存在，导致 l 往右移本来可以让 sum 变大，
        // 但你的代码误以为 sum 减去 nums[0] 后已经太小了，提前退出了循环，
        // 导致左指针被卡死在前面，彻底漏掉了后面更短的优秀解。
        while (sum >= k) {
            res = MMIN(res, r - l + 1);
            sum -= nums[l];
            ++l;
        }
    }

    return res == INF_MAX ? -1 : res;
}
```

### 2. 什么是双指针的“单调性”？

单调性指的是：**当你移动其中一个指针时，区间的某种状态（比如区间的和、区间的长度）只能朝着“一个方向”改变，绝对不会反复横跳。**

以最典型的“寻找和为 $K$ 的子数组（全是正数）”为例：

- **右指针 `r` 向右移**：窗口扩大，区间和**单调递增**（只能变大，不能变小）。
- **左指针 `l` 向右移**：窗口缩小，区间和**单调递减**（只能变小，不能变大）。

### 3. 为什么双指针算法一定要有单调性？

双指针算法之所以能把 $O(N^2)$ 的暴力枚举优化到 $O(N)$，本质上是因为它**利用单调性，安全地跳过了绝大多数不可能成为答案的组合（即高效剪枝）**。

#### 正常剪枝逻辑（数组全为正数）

假设当前窗口是 `[l, r]`，此时满足 `sum >= K`：

1. **已知前提**：因为全是正数，如果固定 `l`，继续把 `r` 往右移，得到的全新和一定会更大。
2. **得出结论**：后面的 `[l, r+1]`, `[l, r+2]` 虽然也满足 `>= K`，但它们的长度一定比当前的 `[l, r]` 更长，**绝对不可能是“最短子数组”**。
3. **安全剪枝**：我们不需要去计算 `[l, r+1]` 之后的组合了。我们可以放心让 `l` 往右移动（`++l`），直接把以当前 `l` 开头的所有后续可能性全部过滤掉。

> **单调性给了算法做出“正确决策”的绝对依据。指针只能向前，绝不回头。**

### 4. 如果没有单调性，会发生什么？

一旦数组里有了负数，单调性就破产了：

- `r` 往右走遇到负数，`sum` 可能会**变小**。
- `l` 往右走移出负数，`sum` 可能会**变大**。

此时，当 `sum < K` 时，算法根本不敢随意做出决策：

- 按照之前的滑动窗口逻辑，`sum < K` 说明窗口和不够大，应该停止移动 `l`，转而去右移 `r`。
- **但实际情况是**：可能 `l` 只要再往右挪一步（移出一个很大的负数），`sum` 就会瞬间暴涨并重新满足 `>= K`，并且这个新窗口的长度更短、更优秀！

#### 总结

没有了单调性，双指针的每一次移动都是在“赌博”。只要赌输一次（错过了一个潜在的可能），整个算法就彻底算错了。

要想在有负数的情况下不错过正确答案，常规的双指针必须频繁回溯指针，这会导致时间复杂度退化为 $O(N^2)$。因此，解决本题需要借用“前缀和 + 单调队列”来强行手动维护一种单调性。
